## 2.1 输入规则

状态：已确认

一、变量选择

1. 用户通过“搜索框 + 分类浏览”选择变量。
2. 搜索框是主要入口，分类浏览用于帮助不清楚变量准确名称的用户查找。
3. 搜索应支持：

   * 变量名称；
   * 变量符号；
   * 变量唯一 ID；
   * 后续中文界面中的中文名称。
4. 分类可按照车辆、轮胎、动力系统、道路、空气动力学等模块组织。

二、变量输入

1. 用户选择变量后，输入数值并选择对应单位。
2. 支持不同变量使用不同单位混合输入。
3. 所有输入在进入公式计算前统一转换为内部 SI 单位。
4. 每个变量只能保留一个用户输入值。
5. 用户重复添加同一变量时：

   * 不新建第二个输入项；
   * 自动定位并高亮已有输入项；
   * 允许用户直接修改原有数值或单位。

三、输入项编辑

1. 已添加的输入项可以直接修改：

   * 数值；
   * 单位；
   * 删除。
2. 用户切换单位时，系统自动换算当前数字，使物理量保持不变。
3. 例如：

   * 200 ft·lb 切换为 N·m 后，应自动显示约 271.16 N·m；
   * 不能只改变单位标签而保持数字不变。
4. 支持删除单个输入。
5. 支持清空全部输入。
6. 清空全部输入前需要用户确认。

四、计算触发原则

1. 第一次计算由用户点击 Calculate 按钮触发。
2. 输入、修改或删除参数后，系统不立即自动重新计算。
3. 受影响的旧结果应标记为过期，并提示需要重新计算。
4. 按钮在已有结果需要更新时显示为 Recalculate。
5. 旧结果失效和重新计算的完整规则在“2.2 默认值与计算触发规则”中进一步确定。

五、允许输入可推导变量

1. 用户可以直接输入本来能够通过其他公式推导出的变量。
2. 用户输入值不得被公式推导结果静默覆盖。
3. 系统应同时保留：

   * 用户输入值；
   * 公式推导值。
4. 如果两者不同，应在旁边显示：

   * 推导值；
   * 绝对差值；
   * 百分比差值；
   * 使用的公式或推导路径。
5. 用户输入值和公式推导值必须标记不同来源。

六、数值来源

系统内部的每个数值至少区分以下来源：

1. User Input：用户输入；
2. Derived：公式推导；
3. Assumed：用户明确启用的系统假设；
4. Constant：系统固定常数。

七、输入有效性

以下情况不得加入计算：

1. 空值；
2. 非数字；
3. 数学上无效的数值；
4. 明确物理不可能的数值；
5. 会直接导致公式无效的数值，例如必要分母为零。

以下情况允许加入，但需要提示：

1. 超出普通车辆常见范围、但现实中可能存在的数值；
2. 赛车、卡车、大型 SUV 或特殊车辆可能使用的参数；
3. 疑似选择了错误单位的数值。

异常但可能的值不得使用阻断式弹窗误导用户，应在页面内显示非阻断警告，并明确说明计算仍可继续。

八、单位误用建议

1. 当某个数值在当前单位下明显异常时，系统应尝试使用该变量支持的其他常见单位解释相同数字。
2. 如果换用另一单位后数值进入合理范围，应向用户提出建议。
3. 例如：
   Wheel radius = 13.8 ft appears unusually large.
   Did you mean 13.8 in?
4. 系统不得未经用户确认自动修改单位。
5. 应提供“一键采用建议单位”的操作。
6. 用户也可以忽略建议并保留原输入。

九、快捷输入

1. v0.1 保留轮胎规格快捷输入。
2. 用户可以按照标准轮胎代码输入，例如：
   195 / 55 R16
3. 快捷输入最终拆分为标准变量：

   * 轮胎断面宽度；
   * 扁平比；
   * 轮辋直径。
4. 快捷输入只是标准变量的便捷填写方式，不建立独立于变量库之外的计算逻辑。
5. 后续如果其他参数同时满足以下条件，也可以增加快捷输入：

   * 使用频率高；
   * 有标准格式；
   * 一次输入代表多个变量；
   * 能明显减少用户操作。

十、当前阶段不包含

v0.1 暂不优先实现：

1. 文件导入；
2. 批量粘贴变量；
3. OCR 或题目截图识别；
4. 用户账号同步；
5. 云端保存；
6. 输入历史记录。

## 2.2 默认值与计算触发规则

状态：已确认

一、系统常数

以下内容属于固定系统常数，由系统自动使用：

1. 标准重力加速度 g = 9.80665 m/s²；
2. 数学常数，例如 π；
3. 已确认的单位换算常数。

系统常数不属于用户输入，也不属于简化假设。用户可以在推导详情中查看，但一般不能修改。

二、不得自动猜测的参数

系统不得根据典型值、常见范围或经验值，自动填写车辆和工况参数，包括但不限于：

1. 车辆重量或质量；
2. 发动机扭矩、功率和转速；
3. 传动比和传动效率；
4. 摩擦系数；
5. 轴距、质心位置和质心高度；
6. 轮胎尺寸；
7. 空气动力学参数；
8. 其他车辆结构和工况参数。

除明确规定的简化假设外，缺少参数时必须提示用户缺少什么，不得静默补值。

三、默认简化假设

为提高基础计算的可用性，v0.1 默认启用以下简化假设：

1. θ = 0：默认道路为平路；
2. R_hx = 0：默认不存在挂车或挂钩水平力；
3. D_A = 0：默认忽略空气阻力；
4. R_x = 0：默认忽略滚动阻力。

这些假设只能在对应公式允许该简化条件时使用，不能任意用于其他公式。

当某条公式允许这些默认简化假设，并且对应假设处于启用状态时，相应变量视为已经由 Assumed 来源提供，系统不得再将这些变量提示为缺失条件。

四、假设必须可见

1. 默认假设不能在后台隐藏使用。
2. 系统只需要突出显示当前计算实际使用到的假设。
3. 结果区域应说明：

   * 使用了多少项假设；
   * 使用了哪些假设；
   * 哪些结果受到这些假设影响。
4. 推导详情中应显示：

   * 假设变量；
   * 假设数值；
   * 假设的物理含义；
   * 假设来源标记 Assumed。

五、假设控制

用户应能够：

1. 单独关闭某一项默认假设；
2. 一键关闭全部默认假设；
3. 恢复默认简化假设。

某项假设关闭后，如果没有其他有效来源提供该变量，相关公式不得继续计算，系统应提示缺少的条件。

六、真实条件替代假设

当某个变量当前使用假设值，而用户输入或其他公式提供了真实值时：

1. 真实值自动替代对应假设；
2. 原假设不再参与计算；
3. 系统提示假设已被可用值替代；
4. 受影响的旧结果标记为需要重新计算。

即使用户输入值与假设值数值相同，也应使用用户输入替代假设，因为数值来源已经发生变化。

七、数值来源优先关系

对于替代简化假设，优先级为：

1. 用户输入；
2. 公式推导结果；
3. 简化假设。

用户输入值和公式推导值同时存在时，不静默互相覆盖，应按照“多路径结果规则”保留并比较；但两者都优先于简化假设。

八、第一次计算

1. 用户添加完已知量后，点击 Calculate 按钮开始计算。
2. 系统不在用户输入每一个字符时自动计算。
3. 没有有效输入时，Calculate 按钮不可用。
4. 用户正在输入尚未完成的数字时，例如只有负号或小数点，不立即报错。

九、输入变化后的处理

以下变化会改变计算条件：

1. 修改输入数值；
2. 添加或删除输入；
3. 接受单位误用建议，使物理值发生变化；
4. 开启或关闭简化假设；
5. 用户输入或推导值替代原有假设。

发生这些变化后，系统不立即重新计算，而是：

1. 根据公式依赖关系找出受影响的结果；
2. 只将直接和间接受影响的结果标记为过期；
3. 不受影响的结果继续保持有效；
4. 按钮由 Calculate 或最新状态变为 Recalculate。

十、过期结果

在用户点击 Recalculate 前，过期结果暂时保留，便于用户查看修改前的数值，但必须：

1. 使用明显的失效样式，例如数值变红；
2. 标记 Needs recalculation；
3. 不再被视为当前有效结果；
4. 不得继续作为后续公式的有效输入。

十一、不需要重新计算的单位切换

如果用户只切换显示单位，并且系统同步换算数值，使物理量保持不变，例如：

200 ft·lb → 271.16 N·m

则：

1. 不标记结果过期；
2. 不需要重新计算；
3. 只更新显示数值和单位。

如果用户接受单位误用建议，例如：

13.8 ft → 13.8 in

由于物理值发生变化，相关结果必须标记为过期并重新计算。

十二、添加新输入

1. 新输入如果与现有结果无关，现有结果继续有效。
2. 新输入可能解锁新的公式和结果时，系统提示可以重新计算。
3. 新输入如果对应一个已经推导出的变量，应保留用户输入值与原推导值，并在重新计算后进行比较。
4. 依赖该变量的后续结果应标记为需要重新计算。

十三、删除输入

删除输入后：

1. 所有直接和间接依赖该输入的结果标记为过期；
2. 系统不得自动恢复以前被替代的假设；
3. 用户点击 Recalculate 后重新扫描公式库；
4. 仍可通过其他路径得到的结果重新计算；
5. 已经无法计算的旧结果从当前有效结果区移除；
6. 系统明确提示这些结果现在缺少哪些条件；
7. 缺失条件区域说明用户补充什么参数后可以恢复计算。

十四、重新计算

用户点击 Recalculate 后，系统应：

1. 使用当前全部有效输入、推导值、假设和系统常数重新计算；
2. 更新所有直接和间接依赖结果；
3. 清除已经解决的过期标记；
4. 更新推导路径；
5. 更新缺失条件；
6. 更新警告、差异和假设说明；
7. 移除当前条件下已经无法得到的旧结果；
8. 对无法得到的结果提示缺少的条件。

十五、按钮状态

按钮状态原则如下：

1. 没有有效输入：
   Calculate 不可用；
2. 有有效输入、尚未计算：
   Calculate；
3. 结果为最新状态：
   显示计算完成状态；
4. 输入或假设发生变化：
   Recalculate；
5. 正在计算：
   Calculating…；
6. 计算无法完成：
   显示具体原因，并允许用户修改条件后再次计算。

## 2.3 警告和合理性规则

状态：已确认

一、检查范围

系统应检查：

1. 用户输入值；
2. 公式推导结果；
3. 输入单位是否可能选择错误；
4. 数学定义域是否满足；
5. 物理条件是否有效；
6. 公式适用条件是否满足；
7. 当前计算使用了哪些简化假设。

同一变量通过不同路径得到多个结果时的比较规则，在“多路径结果规则”中单独规定。

二、合理性状态分级

系统将数值和计算状态分为以下四类：

1. 正常；
2. 异常但可能；
3. 明显不合理；
4. 无效或无法计算。

不同类别必须采用不同的处理方式，不能使用相同的错误提示。

三、正常

当数值、单位、数学条件和物理条件均处于合理范围内时：

1. 正常参与计算；
2. 不显示警告；
3. 可以继续参与后续递归推导。

四、异常但可能

异常但可能，是指数值超出普通乘用车或常见工况的典型范围，但在赛车、卡车、大型 SUV、特殊车辆或特殊工况中仍可能存在。

例如：

1. 23 in 轮辋；
2. 较高的发动机转速；
3. 较大的发动机功率；
4. 大型车辆的重量或轮胎尺寸。

处理规则：

1. 允许用户保留该值；
2. 允许继续计算；
3. 在对应输入或结果旁显示非阻断警告；
4. 明确说明该数值超出常见范围，但可能有效；
5. 明确说明计算仍将继续；
6. 不使用容易让用户误以为计算被禁止的阻断式弹窗。

五、明显不合理数值的检查顺序

当数值在当前单位下明显不合理时，系统不得立即只提示“数值错误”，而应先检查是否可能选择了错误单位。

检查顺序如下：

1. 确认输入是有效数字；
2. 确认该数值是否违反绝对数学或物理限制；
3. 如果没有违反绝对限制，但在当前单位下明显不合理，则检查该变量支持的其他同维度常见单位；
4. 保持输入数字不变，分别使用其他候选单位重新解释；
5. 判断是否有其他单位能够使数值进入正常或合理范围。

例如：

Wheel radius = 13.8 ft

系统检查：

13.8 in ≈ 0.3505 m

如果 13.8 in 属于合理轮胎半径，应提示用户可能选错了单位。

六、存在合理候选单位时

当相同数字使用另一支持单位后进入合理范围时，系统应显示：

1. 当前输入值和单位；
2. 为什么当前单位下明显异常；
3. 推荐的候选单位；
4. 使用候选单位后对应的物理值；
5. 一键采用推荐单位的操作；
6. 保留原输入的操作。

示例：

13.8 ft is highly unusual for a wheel radius.
Did you mean 13.8 in?

用户可以选择：

1. Use 13.8 in；
2. Keep 13.8 ft。

系统不得未经用户确认自动修改单位。

七、不存在合理候选单位时

如果该数值在所有支持单位下都不合理，则：

1. 不再提供单位更改建议；
2. 只提示当前数值明显不合理；
3. 说明常见或合理范围；
4. 建议用户检查输入值、车辆类型和数据来源；
5. 允许用户明确确认后继续使用，除非该值属于绝对无效输入。

八、用户确认保留明显异常值

当用户选择保留明显异常值时：

1. 该值可以参与计算；
2. 警告继续保留；
3. 警告标记为 User confirmed；
4. 系统不得因为用户确认而隐藏警告；
5. 依赖该值的结果也应保留异常来源提示；
6. 用户修改该值后，应重新进行合理性检查；
7. 重新计算时，应重新验证该异常值。

九、无效输入

以下情况属于无效输入，不允许参与计算：

1. 空值；
2. 非数字；
3. NaN；
4. 无穷大；
5. 明确违反变量物理定义的数值；
6. 明确违反数学定义域的数值；
7. 必要分母为零；
8. 不允许出现负数的物理量输入负值；
9. 效率等有明确绝对范围的变量超出物理范围；
10. 其他会使公式无法合法执行的输入。

无效输入不得通过“用户确认继续”绕过。

十、无效输入的影响范围

一个输入无效时，只阻止依赖该输入的计算分支，不应停止整个系统。

例如：

1. 轮胎宽度无效：

   * 不能计算轮胎半径；
   * 不能计算依赖轮胎半径的车速和牵引力；
2. 但如果发动机扭矩和转速有效：

   * 仍可计算发动机功率。

系统应继续计算所有与无效输入无关的可用结果。

十一、公式计算无效

如果某条公式因为数学条件或适用条件不满足而无法执行：

1. 不生成伪造结果；
2. 不使用 NaN、Infinity 或空白数值继续推导；
3. 只停止该公式及依赖该结果的后续分支；
4. 其他独立分支继续计算；
5. 明确说明无法计算的具体原因；
6. 如果存在其他可行公式路径，应继续尝试其他路径。

十二、推导结果异常

如果公式可以正常执行，但推导结果超出典型工程范围：

1. 结果仍然显示；
2. 不自动修改结果；
3. 在结果旁显示合理性警告；
4. 显示结果来源公式；
5. 提醒用户检查输入、单位、假设和适用条件；
6. 该结果仍可继续推导，但其异常状态应传递到依赖结果中。

如果结果在数学或物理上无效，则不得继续参与后续推导。

十三、警告必须指向具体对象

每条警告应明确说明：

1. 对应的变量、输入或结果；
2. 当前数值和单位；
3. 警告原因；
4. 是否继续计算；
5. 用户可以采取的操作；
6. 是否存在建议单位；
7. 哪些结果受到影响。

不得只显示笼统信息，例如：

Invalid input.

应显示具体说明，例如：

Drivetrain efficiency = 1.2 exceeds the physical upper limit of 1.0 and cannot be used.

十四、警告显示方式

1. 警告应显示在相关输入或结果附近；
2. 页面可以同时设置统一的警告汇总区域；
3. 异常但可能的值使用非阻断提醒；
4. 明显不合理的值使用强警告；
5. 无效输入使用错误提示；
6. 不依赖阻断式浏览器弹窗作为主要警告方式；
7. 具体颜色、图标和页面布局在“界面与交互”阶段确定。

十五、检查时机

输入初步检查在以下时机进行：

1. 用户按 Enter；
2. 输入框失去焦点；
3. 用户离开当前输入项；
4. 用户主动添加该输入。

此时检查：

1. 是否为有效数字；
2. 是否属于绝对无效输入；
3. 是否明显超出合理范围；
4. 是否可能选择了错误单位。

完整检查在用户点击 Calculate 或 Recalculate 时进行，包括：

1. 全部输入合理性；
2. 数学定义域；
3. 公式适用条件；
4. 单位误用；
5. 推导结果合理性；
6. 当前假设；
7. 受影响的计算分支。

系统不应在用户每输入一个字符时持续弹出警告。

十六、警告的更新和消除

1. 用户修改输入后，系统重新检查该值；
2. 数值恢复正常后，对应警告自动消失；
3. 用户接受建议单位后，重新检查新值；
4. 用户确认保留异常值后，警告保留并标记 User confirmed；
5. 用户改变已确认异常值后，旧确认失效，必须重新检查；
6. 每次 Recalculate 时，重新检查所有当前警告。

十七、与简化假设的关系

简化假设本身不属于错误，但必须显示。

例如：

Aerodynamic drag is assumed to be zero.

如果用户输入或公式推导出真实值：

1. 真实值替代假设；
2. 原假设提示移除；
3. 受影响结果标记为需要重新计算；
4. 重新计算后更新警告和推导路径。

十八、具体技术阈值

本节点只确定产品行为原则。

以下内容在“单位与工程安全”阶段进一步确定：

1. 每个变量的正常范围；
2. 异常但可能范围；
3. 明显不合理范围；
4. 绝对无效范围；
5. 候选单位匹配算法；
6. 多个候选单位的排序方式；
7. 具体数值阈值和容差。

## 2.4 多路径结果规则

状态：已确认

一、多路径的定义

多路径是指同一个物理变量同时存在多个数值来源，包括：

1. 用户直接输入；
2. 一条公式推导；
3. 多条不同公式推导；
4. 不同物理模型分别计算；
5. 系统简化假设；
6. 系统固定常数。

系统不得仅保存一个数值并覆盖其他来源，必须保留各个来源及其计算路径。

二、数值来源分类

每个数值必须标记来源：

1. User Input：用户直接输入；
2. Derived：公式推导结果；
3. Assumed：简化假设；
4. Constant：系统固定常数。

推导结果还必须记录：

1. 使用的公式；
2. 使用的原始输入；
3. 使用的中间变量；
4. 使用的简化假设；
5. 所属物理模型；
6. 当前是否被选为后续计算使用值。

三、禁止静默覆盖

1. 后生成的结果不得静默覆盖已经存在的数值。
2. 用户输入不得被推导结果自动替换。
3. 一条推导路径不得覆盖另一条推导路径。
4. 不同物理模型的结果必须分别保存。
5. 所有来源均应能够在结果详情中追溯。

四、用户输入与推导结果

当用户直接输入一个本来也能够通过公式推导的变量时：

1. 用户输入值默认作为当前有效值参与后续计算；
2. 系统仍然执行可用公式，得到推导值用于核对；
3. 推导值不得自动覆盖用户输入值；
4. 系统应显示：

   * 用户输入值；
   * 推导值；
   * 绝对差值；
   * 百分比差值；
   * 推导公式或路径；
5. 如果两者一致，应标记为一致或已验证；
6. 如果两者不一致，应按照差异程度显示提示或警告；
7. 用户可以主动选择改用推导值；
8. 用户切换使用值后，依赖该变量的后续结果标记为需要重新计算。

五、同一物理模型下的多条路径

如果多条独立公式描述的是相同条件和相同物理模型，则它们的结果可以相互验证。

如果结果在允许容差内一致：

1. 系统可以将其显示为一个主要结果；
2. 标记该结果已被多条独立路径验证；
3. 详情中保留每一条路径的原始结果；
4. 不得删除各路径来源；
5. 不得把重复执行同一公式算作多路径验证。

主要显示值优先采用公式库中指定的主要路径结果，不默认将不同路径结果取平均。具体主要路径和数值容差在后续公式系统及工程安全阶段确定。

六、同一物理模型下的结果冲突

如果多条本应描述相同物理条件的独立路径，计算结果差异超过允许容差，则标记为：

Calculation conflict

处理规则：

1. 所有冲突结果均保留；
2. 显示每条路径的结果；
3. 显示绝对差值和百分比差值；
4. 显示每条路径使用的公式、输入和假设；
5. 系统不得自动选择其中一个作为正确答案；
6. 依赖该变量的后续计算分支暂停；
7. 与该变量无关的其他计算分支继续运行；
8. 用户修正输入或明确选择使用某一条路径后，该分支才可以继续；
9. 用户选择路径后，相关后续结果标记为需要重新计算；
10. 冲突记录继续保留，直到冲突原因被解决。

七、不同物理模型的结果

如果同一个变量由不同物理模型计算得到，结果不同不自动视为计算冲突。

例如：

1. 发动机牵引力限制模型得到的加速度；
2. 理想恒功率模型得到的加速度。

系统应：

1. 分别显示每个模型结果；
2. 明确标注模型名称；
3. 显示每个模型的假设和适用条件；
4. 标记为 Different models，而不是 Calculation conflict；
5. 不将不同模型结果自动合并；
6. 不自动判断哪个模型结果正确；
7. 不允许后续公式在没有明确选择的情况下任意使用其中一个结果。

八、不同模型的后续计算

当后续公式需要使用一个存在多个模型结果的变量时：

1. 用户应明确选择使用哪个模型结果；
2. 系统也可以根据已明确的公式适用条件选择模型，但必须说明原因；
3. 未选择模型时，依赖该变量的后续分支暂不计算；
4. 用户选择模型后，只影响依赖该变量的计算分支；
5. 切换模型后，相关后续结果标记为需要重新计算；
6. 其他独立结果不受影响。

九、简化假设与真实值

简化假设的优先级低于用户输入和公式推导结果。

例如当前存在：

D_A = 0 N — Assumed

如果用户输入或公式推导得到真实的 D_A：

1. 真实值自动替代简化假设；
2. 假设值退出当前有效计算；
3. 假设值不再作为一个平等的多路径结果展示；
4. 系统应记录该假设曾被替代；
5. 依赖该变量的后续结果标记为需要重新计算。

十、数值使用优先原则

默认使用规则如下：

1. 用户输入值优先作为后续计算使用值；
2. 公式推导值用于核对用户输入；
3. 没有用户输入时，使用公式库指定的主要推导路径；
4. 多条同模型路径冲突时，不自动选值；
5. 不同模型并存时，不自动选值；
6. 简化假设只在没有用户输入和可用推导值时使用；
7. 系统常数按照固定定义使用。

十一、独立路径判定

只有真正独立的计算路径才能用于相互验证。

系统必须追踪每条结果依赖的原始输入和中间结果，防止把以下情况误认为独立验证：

1. 同一公式重复执行；
2. 同一组输入经过等价公式重复计算；
3. 正向计算后再使用该结果反向计算；
4. 结果通过循环依赖回到原变量；
5. 一条路径完全依赖另一条路径的输出。

例如：

发动机转速
→ 使用 F202 计算车速
→ 使用该车速通过 F203 反算发动机转速

反算结果依赖原发动机转速，不能声称发动机转速已被独立路径验证。

十二、循环和自我验证

系统必须防止：

1. 公式循环无限执行；
2. 同一结果反复生成；
3. 由自身间接推导自身；
4. 自我验证被标记为独立验证；
5. 同一冲突警告重复出现。

一条公式使用相同输入、相同单位和相同假设重复执行时，只保留一个结果记录。

十三、结果状态

同一变量的结果可以具有以下状态：

1. Active：当前被后续计算使用；
2. Alternative：可选路径结果；
3. Verified：被独立路径验证；
4. Conflict：与同模型路径发生冲突；
5. Different model：来自不同物理模型；
6. Stale：输入变化后需要重新计算；
7. Invalid：该路径无法合法计算；
8. Assumed：来自简化假设。

十四、重新计算

点击 Recalculate 后：

1. 根据当前输入、假设和用户路径选择重新计算；
2. 更新全部路径结果；
3. 重新判断路径是否独立；
4. 重新比较用户值和推导值；
5. 重新判断同模型路径是否一致；
6. 更新不同模型结果；
7. 更新当前 Active 值；
8. 更新依赖分支；
9. 清除已经解决的冲突；
10. 保留仍未解决的冲突和模型差异。

十五、具体容差

本节点只确定多路径结果的产品处理原则。

以下内容在“单位与工程安全”阶段进一步确定：

1. 数值一致的绝对容差；
2. 数值一致的相对容差；
3. 不同变量应采用的容差类型；
4. 浮点计算误差；
5. 单位换算和显示舍入误差；
6. 冲突严重程度的技术阈值。



多路径结果规则补充

一、复合路径状态的优先级

当同一个变量存在三条或更多同模型路径，且部分路径一致、部分路径冲突时：

1. 只要存在任何尚未解决的同模型路径冲突，该变量的整体状态必须为 Conflict；
2. Conflict 的整体状态优先于 Verified；
3. 不得因为其中两条路径相互一致，就将变量整体标记为 Verified；
4. 结果详情中可以说明哪些路径之间一致；
5. 只有全部有效同模型路径均在允许容差内一致时，整体状态才可标记为 Verified。

示例：

* 路径 A 与路径 B 一致；
* 路径 C 与路径 A、B 冲突；

则整体状态为 Conflict。详情中可以显示“A 与 B 相互验证”，但主状态不得显示为 Verified。

二、主要路径的指定

1. 当同一个变量存在多条同模型推导路径时，主要路径必须由公式系统明确指定；
2. 公式系统阶段应在 FORMULAS.md 中增加主要路径或优先级字段，例如：

   * is_primary；
   * priority；
   * preferred_path；
3. 在主要路径尚未正式指定前，不得使用“最先计算完成的结果”作为主要结果，因为计算顺序属于实现细节，可能发生变化；
4. 临时需要选择代表值时，应采用固定且可重复的规则，例如按照 Formula ID 顺序选择；
5. 临时代表值只用于显示，不代表该路径在工程上更权威；
6. 主要路径正式确定后，应以 FORMULAS.md 的定义为准。

三、不同物理模型的推荐机制

不同物理模型的结果并存时，系统不必在所有情况下都强制用户手动选择。

规则如下：

1. 公式系统可以为某个变量指定 Recommended model；
2. 当推荐模型的适用条件明确满足时，系统可以自动将其设为 Active；
3. 其他模型结果继续作为 Alternative 展示；
4. 系统必须明确显示：

   * 当前使用的模型；
   * 为什么推荐该模型；
   * 其他可选模型；
   * 各模型的适用条件和假设；
5. 用户可以主动切换到其他模型；
6. 用户切换模型后，依赖该变量的后续结果标记为需要重新计算；
7. 如果没有指定推荐模型；
8. 或多个模型均适用且无法自动判断优先级；
9. 或模型的适用条件不足；

则系统不得自行选择，必须要求用户明确选择模型。

对于 v0.1 中可能存在的多个加速模型：

1. 推荐模型、参考模型、Active / Alternative 关系不得在 Part 2 中依赖旧公式编号固定；
2. 具体哪个模型作为推荐模型，应在第三阶段“公式与变量系统”重新确认；
3. Part 2 只要求系统支持：模型名称、模型类型、适用条件、主要假设、Active / Alternative 状态和用户切换机制；
4. 如果某个模型被公式系统标记为推荐模型，并且适用条件满足，系统可以默认将其设为 Active；
5. 其他模型继续作为 Alternative 或 Reference 展示；
6. 用户切换模型后，所有依赖该变量的后续结果进入待重新计算状态；
7. 旧公式编号只能作为 Legacy / Reference 信息，不得作为 Part 2 的固定产品规则。

## 2.5 结果展示规则

状态：已确认

一、结果展示规则的目标

结果展示规则用于规定计算结果如何组织、排序、标记和显示。

系统应保留所有可计算结果，但用户第一眼应优先看到最重要、最可信、最需要注意的结果，而不是被大量中间变量和过程数值淹没。

二、结果展示分层

结果展示区分为三层：

1. 主要结果；
2. 其他推导结果；
3. 中间结果。

主要结果优先显示用户最可能关心的最终结果、用户指定目标、当前被选为 Active 的结果、当前计算链的终点变量，以及有冲突或强警告的关键结果。

其他推导结果显示系统自动得到的其他有效结果，具有工程意义，但视觉层级低于主要结果。

中间结果是具有独立变量符号、独立工程意义，并且可能被其他公式继续使用的变量级结果。中间结果默认折叠，用户可以展开查看。

三、用户输入不重复放入普通结果区

用户输入应主要显示在 Inputs / 已知条件区域。

结果区不应重复列出所有用户输入。只有以下情况，用户输入才需要在结果区出现：

1. 该变量同时存在推导值，需要比较；
2. 用户输入值和推导值不一致；
3. 用户输入值存在警告；
4. 用户输入值被某个结果直接引用，需要解释来源。

四、每个结果的基本信息

每个结果至少应显示：

1. 变量名称；
2. 变量符号；
3. 数值；
4. 单位；
5. 结果状态；
6. 简短来源。

如果存在特殊情况，应附加状态标签，例如：

1. Verified；
2. Conflict；
3. Different model；
4. Alternative；
5. Uses assumptions；
6. Needs recalculation；
7. User confirmed warning；
8. Invalid path。

五、结果状态标签

结果区统一使用以下状态标签：

1. Active：当前用于后续计算的值；
2. Alternative：可选结果，但当前未被选用；
3. Verified：被独立路径验证；
4. Conflict：同模型路径结果冲突；
5. Different model：来自不同物理模型，不算冲突；
6. Uses assumptions：结果使用了简化假设；
7. Warning：数值异常但仍可显示；
8. User confirmed warning：用户确认保留异常值；
9. Needs recalculation：输入变化后旧结果已过期；
10. Invalid path：某条路径无法合法计算。

六、结果排序规则

结果不得按程序计算顺序随意显示。

建议排序顺序为：

1. 冲突结果 / 强警告结果；
2. 用户关心的主要结果；
3. 当前 Active 的终点结果；
4. 不同模型的可选结果；
5. 其他有效推导结果；
6. 中间结果；
7. 无效路径说明。

冲突和强警告结果必须优先显示，避免用户漏看。

七、多路径结果显示

当用户输入值与推导值并存时，主行显示当前 Active 值，并在下方显示：

1. 用户输入值；
2. 推导值；
3. 绝对差值；
4. 百分比差值；
5. 推导路径。

当同模型多路径一致时，显示一个主结果，并标记 Verified by independent paths。各路径的原始结果和推导详情保留在展开区域中。

当同模型多路径冲突时，不显示为普通单一结果，而显示为冲突组，并显示每条路径的结果、公式、差值和后续影响。

当不同物理模型得到同一变量时，不合并、不标记普通冲突，应并列显示各模型结果，并标明 Different physical models。

八、单位显示规则

每个结果默认只突出显示一个主单位。

系统应支持：

1. 用户切换显示单位；
2. 单位切换时自动换算数值；
3. 单纯切换显示单位不触发重新计算；
4. 展开查看 SI 内部值；
5. 展开查看必要的其他单位。

内部 SI 精确值不因显示单位改变。

具体小数位、有效数字和工程显示精度在“单位与工程安全”阶段确定。

九、过期结果展示

输入改变后，受影响结果不立即消失，而是暂时保留旧值并标记 Needs recalculation。

过期结果必须满足：

1. 视觉上明显区别于最新结果；
2. 不再作为当前有效结果；
3. 不参与后续推导；
4. 点击 Recalculate 后更新；
5. 如果重算后无法得到，则从有效结果区移除，并转入缺失条件说明。

十、假设使用展示

如果某个结果使用了简化假设，应在结果旁显示 Uses assumptions 或 Uses N assumptions。

展开后必须列出具体假设，包括：

1. 假设变量；
2. 假设数值；
3. 假设物理含义；
4. 假设来源 Assumed。

十一、警告结果展示

异常但可计算的结果仍然显示，但必须带警告。

如果用户确认保留明显异常值，警告继续保留，并标记 User confirmed warning。

警告不得因为用户确认而消失。

十二、无结果状态

系统不得显示空白页面。

尚未计算时，应提示：

No calculated results yet. Add known values and select Calculate.

已经计算但没有任何可推导结果时，应提示：

No result can currently be calculated. See Missing Conditions for required inputs.

有部分结果但目标结果无法计算时，应提示：

Some results were calculated, but the requested result is not available. See Missing Conditions.

十三、结果对象的定义

结果对象是系统在一次 Calculate 或 Recalculate 后得到的某个变量数值记录。

一个结果对象至少应包含：

1. 变量名称；
2. 变量符号；
3. 内部 SI 数值；
4. 当前显示数值；
5. 当前显示单位；
6. 数值来源；
7. 当前状态；
8. display_role；
9. 使用的公式路径；
10. 依赖的用户输入；
11. 依赖的中间变量；
12. 使用的简化假设；
13. 所属物理模型；
14. 是否为当前 Active 值；
15. 是否可继续用于后续计算。

十四、不作为普通结果展示的内容

以下内容默认不作为普通结果展示：

1. 用户输入值；
2. 系统固定常数；
3. 简化假设；
4. 单位换算过程值；
5. 公式内部临时值；
6. 无效计算路径。

用户输入值默认显示在 Inputs 区域。只有当它需要与推导值比较、存在警告、存在冲突或需要解释来源时，才在结果区域中出现。

系统固定常数、简化假设、单位换算过程和公式内部临时值，默认只在推导详情中显示。

无效计算路径不属于有效结果，应作为 Invalid path 或缺失条件说明展示。

十五、主要结果

主要结果是用户第一眼应该看到的结果。

满足以下任一条件的有效结果，可以归入主要结果：

1. 用户明确指定的目标量；
2. 公式系统标记的重要输出变量；
3. 当前计算图中的有效终点变量；
4. 当前被选为 Active 的模型结果；
5. 存在冲突或强警告、需要优先提醒用户的关键结果。

Conflict 和 Warning 是结果状态，不是结果类型。它们会提高展示优先级，但不改变变量本身的分类。

十六、其他推导结果

其他推导结果是系统自动推导出的有效结果，具有工程意义，但不是当前最核心目标。

其他推导结果应满足：

1. 由公式推导得到；
2. 数学和物理上有效；
3. 不是用户输入的简单重复；
4. 不是公式内部临时量；
5. 具有独立工程含义；
6. 可以被用户查看；
7. 视觉层级低于主要结果。

十七、中间结果

中间结果是具有独立变量符号、独立工程含义，并且可能被其他公式继续使用的变量级结果。

中间结果可以包括质量因子、总传动比、车轮半径、发动机角速度等类型的变量，但这些示例不表示该变量在所有场景下都必须显示为中间结果。

实际显示角色由变量元数据中的 display_role 字段决定。

以下内容不属于中间结果，只属于推导详情或 calculation details：

1. 单位换算后的临时 SI 数值；
2. 公式代入过程中的临时数字；
3. 代码内部临时变量；
4. 无独立工程定义的临时系数；
5. 仅用于显示格式转换的数值；
6. 仅用于某一步算术运算的中间数。

十八、display_role 的来源与存储

display_role 是决定结果展示区域的关键字段。

display_role 的权威来源来自变量元数据，由后续“公式与变量系统”阶段统一定义。

每次 Calculate 或 Recalculate 生成结果对象时，系统应从变量元数据中读取并解析 display_role，然后将解析后的 display_role 写入结果对象。

因此：

1. 变量元数据负责定义该变量默认属于哪类显示角色；
2. 结果对象必须保存本次计算使用的 display_role；
3. 页面展示时优先读取结果对象中的 display_role；
4. 系统不得在展示阶段根据变量名称、公式编号或计算顺序临时猜测分类；
5. 如果变量元数据暂时缺少 display_role，应使用固定、可重复的 fallback 规则，而不是使用代码执行顺序。

十九、display_role 建议取值

display_role 建议取值包括：

1. primary_output；
2. derived_output；
3. intermediate；
4. input_only；
5. constant；
6. assumption。

如果同一个变量在不同上下文下需要不同展示方式，应由后续“公式与变量系统”阶段定义上下文规则，而不是由结果展示规则临时判断。

二十、当前阶段不固定的内容

以下内容不在结果展示规则中固定：

1. 具体颜色；
2. 字体大小；
3. 卡片样式；
4. 动画；
5. 图标；
6. 小数位；
7. 有效数字；
8. 具体工程阈值；
9. 具体公式主路径；
10. 具体推荐模型。

二十一、v0.1 暂不做的结果展示功能

v0.1 暂不做：

1. 图表；
2. 曲线；
3. 多车辆对比；
4. 历史结果保存；
5. 结果收藏；
6. 导出 PDF；
7. 复杂报表；
8. 自定义仪表盘；
9. 云端同步。

## 2.6 缺失条件规则

状态：已确认

一、缺失条件规则的目标

缺失条件规则用于解决两个问题：

1. 系统正向计算后，解释为什么某些结果暂时算不出来；
2. 用户指定目标结果时，反向推导还需要补充哪些条件。

这两件事使用同一套公式依赖图和变量依赖图，不建立两套独立系统。

二、正向推导与反向找条件

系统应支持两种视角：

1. 正向推导：

   * 用户提供已知条件；
   * 系统计算所有当前能得到的结果；
   * 对暂时无法得到的重要结果，说明缺少什么条件。

2. 反向找条件：

   * 用户指定想要求的目标变量；
   * 系统从目标变量反向搜索可用公式路径；
   * 系统列出每条路径还需要哪些条件；
   * 用户据此补充输入。

正向推导和反向找条件本质上是同一张依赖图的不同使用方式。

三、缺失条件的定义

缺失条件不只等于缺少变量输入。

以下情况都可能导致某条路径无法继续计算：

1. Missing：缺少必要变量；
2. Invalid：变量已经提供，但数值无效；
3. Conflict：变量存在同模型多路径冲突；
4. Model not selected：存在多个物理模型，但尚未选择使用哪个；
5. Assumption disabled：某个默认假设被关闭，且没有其他真实值替代；
6. Applicability condition not met：公式适用条件不满足；
7. Circular dependency：路径存在循环依赖；
8. No registered output direction：公式系统没有登记该方向的计算能力。

系统必须区分这些情况，不能统一显示为 Missing input。

四、缺失变量

当某条公式路径需要某个变量，而该变量既没有用户输入，也没有有效推导值、假设值或常数来源时，该变量属于 Missing。

显示时应说明：

1. 缺少的变量名称；
2. 变量符号；
3. 该变量用于哪条公式；
4. 该变量会影响哪个目标结果；
5. 用户可以直接输入该变量，或通过其他路径推导该变量。

五、无效输入

如果用户已经提供某个变量，但该值无效，则不应显示为缺失。

例如：

drivetrain efficiency = 1.2

系统应显示：

Drivetrain efficiency is provided but invalid.
Fix this value before this path can be calculated.

无效输入只阻止依赖该变量的计算分支，不阻止其他独立分支。

六、冲突结果

如果某个变量存在同模型多路径冲突，则不应显示为缺失。

例如：

Vehicle speed is available but unresolved.

系统应显示：

Vehicle speed has a calculation conflict.
Resolve the conflict or choose one path before dependent calculations can continue.

冲突未解决前，依赖该变量的后续分支暂停。

七、模型未选择

如果同一变量存在多个不同物理模型结果，并且没有明确的推荐模型或用户选择，则依赖该变量的后续路径暂不计算。

系统应显示：

Multiple model results are available.
Select one model to continue this calculation.

如果公式系统已经指定推荐模型，并且推荐模型适用条件满足，则系统可以自动使用推荐模型，并将其他模型标记为 Alternative。

八、假设被关闭

如果某个变量原本可以由默认简化假设提供，但用户关闭了该假设，且没有用户输入或推导值替代，则该路径不能继续。

系统应显示：

This condition is missing because the related assumption is disabled.

并提供两个方向：

1. 输入真实变量值；
2. 重新启用该简化假设。

例如：

Road grade angle is missing.
You can either:

* enter road grade angle;
* re-enable flat road assumption.

九、公式适用条件不满足

如果某条公式的变量齐全，但适用条件不满足，则该路径不能计算。

系统不得把这种情况显示为缺少输入。

应显示：

This formula path is not applicable under the current conditions.

并说明：

1. 哪条公式不可用；
2. 不满足哪个适用条件；
3. 是否存在其他公式路径；
4. 用户需要改变什么条件才能使用该路径。

十、反向找条件不是自动代数反解

系统必须区分两件事：

1. 反向找条件；
2. 自动代数反解。

反向找条件是指：

用户想要求某个目标变量，系统查找哪些已登记公式路径可以得到该变量，并说明还缺哪些条件。

自动代数反解是指：

系统自动把公式变形成其他输出方向。

v0.1 必须支持反向找条件，但不允许系统自行对任意公式做自动代数反解。

公式只有在公式系统明确登记某个输出方向时，才允许按该方向计算。

例如：

P = T × ω

公式系统可以登记：

1. 已知 T 和 ω，可求 P；
2. 已知 P 和 ω，可求 T；
3. 已知 P 和 T，可求 ω。

只有登记过的方向，系统才允许计算和反向找条件。

十一、路径方案展示

缺失条件必须按路径方案展示，不能只列散乱变量。

错误展示方式：

Missing:

* vehicle mass
* wheel radius
* engine torque
* drivetrain efficiency
* gear ratio

正确展示方式：

To calculate acceleration, choose one path:

Path 1: Engine torque path
Need:

* engine torque
* total gear ratio
* drivetrain efficiency
* wheel radius
* vehicle mass

Path 2: Power-speed path
Need:

* engine power
* vehicle speed
* vehicle mass

用户看到的应是可补齐方案，而不是无结构变量列表。

十二、路径展开规则

当某条路径缺少中间变量时，系统应继续向下分析该中间变量是否可以通过其他公式推导。

例如：

目标变量需要 wheel radius。

如果 wheel radius 可以由 tire width、aspect ratio 和 rim diameter 推导，则系统应显示：

You can provide wheel radius directly,
or provide tire size parameters:

* tire width
* aspect ratio
* rim diameter

系统不应过早停在中间变量上。

十三、路径排序规则

当存在多条可行路径时，系统应排序展示。

建议排序原则：

1. 缺失条件最少的路径优先；
2. 需要用户手动选择的冲突或模型最少的路径优先；
3. 需要启用特殊假设最少的路径优先；
4. 用户更容易提供的变量优先；
5. 公式系统优先级更高的路径优先；
6. 路径更短、依赖更少的路径优先。

默认显示前 3 条推荐路径，其余路径折叠。

十四、推荐下一步输入

系统可以推荐用户下一步最值得补充的条件。

推荐逻辑包括：

1. 能解锁用户指定目标；
2. 能解锁最多后续结果；
3. 用户通常容易获得该数据；
4. 单位和物理含义清晰；
5. 不依赖复杂判断或特殊测试。

例如：

Recommended next input:
Vehicle mass

推荐输入只是建议，系统不得替用户自动填写数值。

十五、没有指定目标时

如果用户没有指定目标，系统不应对公式库中的所有变量都展开反向缺失分析，否则页面会过载。

系统应：

1. 正向计算所有当前能得到的结果；
2. 对重要输出变量显示缺失条件；
3. 不对所有可能变量显示缺失条件。

重要输出变量由后续公式与变量系统中的 display_role 或 importance 字段决定。

十六、指定目标时

如果用户指定目标变量，则系统应围绕该目标显示缺失条件。

系统应显示：

1. 当前目标；
2. 该目标是否已经能计算；
3. 如果不能计算，列出可行路径；
4. 每条路径缺少什么；
5. 哪些条件已经满足；
6. 哪些条件无效或冲突；
7. 推荐下一步补充什么。

十七、部分结果可得时

如果系统能算出部分结果，但目标结果无法得到，应显示：

Some results were calculated, but the requested result is not available.

并引导用户查看目标结果的缺失条件。

已计算出的无关结果继续正常显示。

十八、循环依赖处理

反向搜索和正向推导都必须防止循环依赖。

如果路径出现循环，例如：

A 需要 B；
B 需要 C；
C 又需要 A；

系统应停止该路径，并显示：

This path contains a circular dependency and cannot be used as an independent solution.

循环路径不得作为有效补齐方案。

十九、缺失条件的作用范围

缺失条件只影响依赖该条件的分支。

如果某个目标无法计算，系统仍应继续显示其他可计算结果。

系统不得因为一个目标缺少条件，就停止整个计算器。

二十、与结果展示规则的关系

缺失条件不属于普通有效结果。

它应显示在单独的 Missing Conditions 区域，或者显示在目标结果卡片的缺失说明中。

结果展示区显示已经得到的有效结果。

缺失条件区显示暂时无法得到的目标、原因和补齐路径。

二十一、与推导展示规则的关系

缺失条件只说明：

1. 缺什么；
2. 为什么缺；
3. 可以通过哪些路径补齐。

具体公式展开、代入过程、单位换算、路径细节，在“推导展示规则”中进一步规定。

二十二、当前阶段不确定的内容

以下内容在后续阶段确定：

1. 每个公式允许哪些输出方向；
2. 每个变量是否容易由用户提供；
3. 每条路径的优先级；
4. 重要输出变量列表；
5. 反向搜索深度限制；
6. 路径推荐算法；
7. 具体界面布局和交互方式。
缺失条件规则补充

一、不同物理模型路径的展示

当用户指定目标变量，且该目标变量可以通过多个不同物理模型路径得到时，系统必须明确标注每条路径所属的物理模型。

不同物理模型路径不得被展示成完全等价的普通输入方案。

每条模型路径至少应显示：

1. 模型名称；
2. 模型类型；
3. 适用条件；
4. 使用的主要假设；
5. 当前已经满足的条件；
6. 当前缺少的条件；
7. 该模型是否为推荐模型；
8. 该模型结果是否仅作为参考结果。

例如：

Path 1: Engine-limited acceleration model
Model type: physical force balance
Assumptions: depends on road load terms and drivetrain force
Missing conditions: ...

Path 2: Ideal constant-power acceleration model
Model type: idealized power-based model
Assumptions: constant available power, simplified drivetrain behavior
Missing conditions: ...

排序规则仍然适用，但模型标签、适用条件和假设说明不得省略。

如果多个模型都能计算同一个目标变量，系统应提醒用户：

These paths use different physical models and may not produce equivalent results.

二、不同模型路径与普通路径的区别

同一物理模型下的不同路径，可以理解为求同一个物理量的不同补齐方案。

不同物理模型下的路径，不只是补齐条件不同，而是模型假设和适用范围不同。

因此：

1. 同模型路径可以用于相互验证；
2. 不同模型路径不得直接用于相互验证；
3. 不同模型路径结果不同，不自动视为计算冲突；
4. 用户需要能看出每条路径背后的物理模型差异；
5. 后续依赖该变量的计算，应使用当前 Active 或推荐模型结果。

三、路径展开深度限制

缺失条件路径可以递归展开。

例如：

目标变量需要某个中间变量；
该中间变量又可以由其他变量推导；
系统可以继续展开该中间变量的补齐路径。

但是递归展开必须设置深度上限，防止：

1. 页面信息过载；
2. 路径无限展开；
3. 循环依赖；
4. 用户难以理解补齐方案；
5. 实现复杂度失控。

展开深度上限由后续“公式与变量系统”阶段确定。

在上限确定前，系统应采用保守策略：

1. 优先展示距离目标最近、缺失条件最少的路径；
2. 默认只展开推荐路径的关键层级；
3. 更深层路径折叠显示；
4. 检测到循环依赖时立即停止该路径；
5. 不得为了展开更多路径而牺牲可读性。

## 2.7 推导展示规则

状态：已确认

一、推导展示规则的目标

推导展示规则用于解释某个结果或某条路径是如何得到的。

它要回答：

1. 这个结果从哪里来；
2. 使用了哪个公式；
3. 使用了哪些用户输入；
4. 使用了哪些推导变量；
5. 使用了哪些系统常数；
6. 使用了哪些简化假设；
7. 是否发生单位换算；
8. 是否存在警告、冲突或过期状态。

推导展示不是展示代码逻辑，而是展示工程计算链条。

二、默认折叠

推导过程默认折叠。

结果展示区只显示结果摘要。用户点击某个结果或某条路径后，才展开对应推导详情。

系统不得默认一次性展开所有结果的完整推导过程，避免页面被大量公式和中间步骤淹没。

三、每个结果单独拥有推导详情

每个可计算结果都应有自己的推导详情。

如果一个变量有多个路径结果，则每条路径都应有独立推导详情。

系统不得只展示当前 Active 路径，而隐藏其他路径的推导来源。

四、推导展示的基本层级

推导详情建议包含以下层级：

1. 推导摘要；
2. 使用公式；
3. 变量代入；
4. 单位换算；
5. 中间变量；
6. 假设与常数；
7. 来源与依赖；
8. 警告、冲突或过期说明。

具体页面布局在“界面与交互”阶段确定。

五、推导摘要

推导详情顶部应先显示一句简短说明，让用户快速理解结果来源。

例如：

Vehicle speed was derived from engine speed, total gear ratio, and wheel radius.

Acceleration was calculated from tractive force, vehicle mass, mass factor, and road load assumptions.

推导摘要应使用工程语言解释计算链条，不应只显示公式编号。

六、公式展示

展开推导后，必须显示使用的公式。

公式展示应包括：

1. 公式表达式；
2. 公式名称；
3. 公式所属模型；
4. 公式适用方向；
5. 后续公式系统确认后的公式 ID；
6. 后续公式系统确认后的来源信息。

Part 2 阶段不固定具体公式编号。公式编号、来源章节和模型归属在后续“公式与变量系统”阶段确认。

七、变量代入

公式下方应显示变量代入过程。

代入过程必须满足：

1. 每个代入值与公式符号对应；
2. 每个代入值带单位；
3. 不显示裸数字；
4. 每个代入值标记来源；
5. 最终结果显示当前用户选择的显示单位；
6. 内部计算仍以 SI 数值为准。

来源标记包括：

1. User Input；
2. Derived；
3. Assumed；
4. Constant。

示例：

V = r_w × ω_e / N_tf

Substitution:
r_w = 0.318 m — Derived
ω_e = 314.16 rad/s — User Input
N_tf = 3.50 — User Input

Result:
V = 28.54 m/s
V = 63.8 mph

八、单位换算展示

单位换算不属于普通结果，但属于推导详情。

如果用户输入单位与内部计算单位不同，推导详情中应显示换算过程。

示例：

Engine torque:
200 ft·lb → 271.16 N·m

Vehicle speed:
60 mph → 26.82 m/s

规则：

1. 用户输入单位必须保留；
2. 内部 SI 数值必须可追溯；
3. 结果显示单位必须可追溯；
4. 单位换算不得改变物理值；
5. 只切换显示单位不触发重新计算；
6. 接受单位误用建议导致物理值改变时，相关结果必须重新计算。

九、假设展示

如果某个结果使用了简化假设，推导详情中必须列出具体假设。

不能只在结果卡片中显示 Uses assumptions，而不说明用了哪些假设。

示例：

Assumptions used:

* road grade angle = 0 — Assumed
* aerodynamic drag = 0 — Assumed
* rolling resistance = 0 — Assumed

每个假设应显示：

1. 假设变量；
2. 假设数值；
3. 假设物理含义；
4. 来源 Assumed；
5. 是否被真实输入或推导值替代。

十、系统常数展示

如果推导使用了系统常数，应在推导详情中显示。

例如：

g = 9.80665 m/s² — Constant
π = 3.14159... — Constant

系统常数不属于用户输入，也不属于简化假设。

十一、中间变量展示

如果推导过程中使用了中间变量，应显示这些中间变量。

中间变量指具有独立变量符号、独立工程意义，并可能被其他公式继续使用的变量结果。

示例：

Intermediate values:

* total gear ratio N_tf = 3.50
* mass factor M_f = 1.18
* wheel radius r_w = 0.318 m
* engine angular speed ω_e = 314.16 rad/s

以下内容不属于中间变量，只属于 calculation details：

1. 单位换算后的临时 SI 数字；
2. 公式代入过程中的临时数字；
3. 代码内部临时变量；
4. 无独立工程定义的临时系数；
5. 仅用于显示格式转换的数值；
6. 仅用于某一步算术运算的中间数。

十二、多路径推导展示

如果同一个变量存在多条路径，系统必须分别展示每条路径的推导。

不得只展示 Active 路径。

十三、同模型多路径一致

如果多条同模型独立路径结果一致，并被标记为 Verified，则推导详情应显示：

1. 每条路径的公式；
2. 每条路径的代入值；
3. 每条路径的结果；
4. 路径之间的差值；
5. 为什么可以视为独立路径；
6. Verified 状态说明。

示例：

Verified by 2 independent paths

Path 1:
Formula ...
Substitution ...
Result ...

Path 2:
Formula ...
Substitution ...
Result ...

十四、同模型多路径冲突

如果多条同模型路径发生冲突，推导详情必须并列显示冲突路径。

不得只显示其中一条路径。

应显示：

1. 每条冲突路径的结果；
2. 每条路径使用的公式；
3. 每条路径依赖的输入；
4. 每条路径使用的假设；
5. 绝对差值；
6. 百分比差值；
7. 冲突对后续计算的影响；
8. 用户可以如何处理。

示例：

Calculation conflict

Path A:
Result = 60 mph
Derivation ...

Path B:
Result = 72 mph
Derivation ...

Difference:
12 mph / 20%

Dependent calculations are paused until this conflict is resolved.

十五、不同物理模型推导展示

如果同一个变量由不同物理模型得到，推导详情必须标记模型差异。

不同物理模型结果不得作为普通冲突展示。

应显示：

1. 模型名称；
2. 模型类型；
3. 模型适用条件；
4. 模型假设；
5. 每个模型的公式；
6. 每个模型的代入过程；
7. 每个模型的结果；
8. 当前 Active 或 Recommended 模型；
9. Alternative 模型。

示例：

Different physical models — not a calculation conflict.

Model 1:
Engine-limited model
Derivation ...

Model 2:
Ideal constant-power model
Derivation ...

具体哪个模型为推荐模型，在后续“公式与变量系统”阶段确定。

十六、过期结果的推导展示

如果结果已经过期，推导详情必须明确标记。

示例：

This derivation is stale.
Inputs have changed since this result was calculated.
Select Recalculate to update this path.

规则：

1. 旧推导可以暂时保留；
2. 必须标记 Needs recalculation；
3. 不得显示为当前有效推导；
4. 不得继续作为后续计算依据；
5. 重新计算后更新推导详情；
6. 如果重算后路径不可用，则转入缺失条件说明。

十七、无效路径的推导展示

如果某条路径无法计算，可以显示未完成推导，但不得伪造结果。

应显示：

1. 该路径使用的公式；
2. 当前已经满足的条件；
3. 缺少或无效的条件；
4. 无法计算的原因；
5. 是否存在替代路径。

示例：

This path cannot be evaluated.

Reason:
wheel radius is missing.

Formula requires:

* wheel radius
* engine speed
* total gear ratio

无效路径不属于有效结果。

十八、反向找条件时的推导展示

当用户指定目标变量，但当前条件不足以计算该目标时，推导展示应显示公式结构、已满足条件和缺失点。

此时不得伪造数值代入过程。

示例：

Target:
Acceleration

Possible path:
Acceleration requires tractive force, vehicle mass, mass factor, and road load terms.

Currently available:

* tractive force
* mass factor

Missing:

* vehicle mass

如果某个缺失中间变量还能继续展开，则显示其补齐路径。

递归展开深度由后续“公式与变量系统”阶段确定。

十九、推导链展示形式

v0.1 建议采用树状结构或步骤列表展示推导链。

示例：

Acceleration
└─ Formula: ...
├─ Tractive force F_x
│  └─ Derived from engine torque and gear ratio
├─ Vehicle mass M
│  └─ User input
├─ Mass factor M_f
│  └─ Derived from total gear ratio
└─ Road load terms
└─ Assumed zero

v0.1 不要求复杂图谱。

二十、推导必须可追溯

每条推导详情至少应能追溯到：

1. 原始用户输入；
2. 中间变量；
3. 使用公式；
4. 简化假设；
5. 系统常数；
6. 单位换算；
7. 最终结果；
8. 警告或冲突状态。

系统不得只显示最终公式和最终结果。

二十一、禁止展示代码内部逻辑

普通推导展示不应显示：

1. JavaScript 函数名；
2. 对象 key；
3. 数组索引；
4. 递归搜索内部状态；
5. 调试日志；
6. 代码变量名；
7. 浮点误差调试细节。

这些内容只可在未来 developer/debug mode 中考虑。

普通用户需要看到的是工程推导，不是程序执行过程。

二十二、精度与舍入

推导展示中的数值可以按显示精度舍入。

但必须遵守：

1. 内部计算不得使用显示舍入值继续计算；
2. 如果因舍入导致代入式看起来略有差异，应说明显示值已舍入；
3. 具体小数位、有效数字和工程显示精度，在“单位与工程安全”阶段确定。

二十三、与结果展示规则的关系

结果展示规则负责决定：

1. 哪些结果显示在主结果区；
2. 哪些结果显示在其他推导结果区；
3. 哪些结果显示在中间结果区；
4. 结果如何排序；
5. 状态如何标记。

推导展示规则负责说明：

1. 某个结果怎么来的；
2. 某条路径怎么来的；
3. 为什么某条路径有效或无效；
4. 使用了哪些公式、输入、假设和单位换算。

二十四、与缺失条件规则的关系

缺失条件规则负责说明：

1. 目标暂时算不出来；
2. 缺什么；
3. 可通过哪些路径补齐。

推导展示规则负责展开这些路径的公式结构、已满足条件和缺失点。

二十五、v0.1 暂不做

v0.1 暂不要求：

1. 自动生成完整 LaTeX 报告；
2. 导出 PDF；
3. 动态图形推导图；
4. 复杂交互图谱；
5. 自动证明公式；
6. 自动代数变形过程展示；
7. 多语言推导切换；
8. developer/debug mode。

v0.1 只要求把每个结果的公式、代入、单位、来源、假设和路径关系讲清楚。
推导展示规则补充

一、中间变量示例不得作为硬编码分类

推导展示规则中列出的中间变量示例，只用于说明这类变量通常具有以下特征：

1. 有独立变量符号；
2. 有独立工程意义；
3. 可以被其他公式继续使用；
4. 可能出现在推导链中。

这些示例不表示该变量在所有场景下都必须显示为中间结果。

例如：

1. total gear ratio N_tf 如果由用户直接输入，则它首先属于用户输入；
2. engine angular speed ω_e 如果是用户关心的推导结果，也可以显示为其他推导结果；
3. wheel radius r_w 如果是用户指定目标，也可以成为主要结果；
4. mass factor M_f 如果只是服务于加速度计算，通常可显示为中间结果。

实际显示角色由变量元数据中的 display_role 字段决定。

推导展示规则不得把示例变量硬编码为固定分类。

二、display_role 优先

当变量的显示角色存在不确定性时，应优先遵循变量元数据中的 display_role。

display_role 可将变量定义为：

1. primary_output；
2. derived_output；
3. intermediate；
4. input_only；
5. constant；
6. assumption。

如果同一个变量在不同上下文下需要不同展示方式，应由后续“公式与变量系统”阶段定义上下文规则，而不是由推导展示规则临时判断。

## 2.8 v0.1 精确验收范围

状态：已确认

一、v0.1 的定位

v0.1 的目标不是覆盖大量公式，也不是一次性实现完整车辆动力学公式册。

v0.1 的目标是完成一个最小但完整的纵向闭环：

输入变量
→ 单位转换
→ 正向推导
→ 反向找条件
→ 缺失条件提示
→ 多路径处理
→ 警告与合理性检查
→ 结果分层展示
→ 推导展开
→ 输入修改后的重新计算机制

v0.1 应证明产品规则、数据结构和推导架构是成立的。

二、v0.1 必须包含的输入能力

v0.1 必须支持：

1. 搜索变量；
2. 分类浏览变量；
3. 添加输入；
4. 修改输入；
5. 删除输入；
6. 清空全部输入；
7. 每个变量只能保留一个用户输入值；
8. 支持不同变量混合单位输入；
9. 单位切换时自动换算数值，保持物理量不变；
10. 疑似单位错误时提示，不自动修改；
11. 用户确认后才能采用建议单位；
12. 用户输入本来也能推导出的变量时，保留用户值与推导值对比。

三、v0.1 必须包含的单位能力

v0.1 必须支持：

1. 用户输入单位；
2. 内部统一 SI 计算；
3. 结果使用用户友好单位显示；
4. 显示单位切换；
5. 单位切换不触发重新计算；
6. 接受单位误用建议后，相关结果进入待重新计算状态；
7. 推导详情中显示用户单位到 SI 的换算过程；
8. 推导详情中显示 SI 到显示单位的换算过程。

v0.1 不要求覆盖所有工程单位，但必须覆盖当前 v0.1 公式模块所需的单位。

四、v0.1 必须包含的正向推导能力

v0.1 必须支持：

1. 用户输入一组已知条件；
2. 系统自动推导当前所有可得结果；
3. 多步递归推导；
4. 防止循环推导；
5. 防止重复路径无限执行；
6. 不生成 NaN；
7. 不生成 Infinity；
8. 不生成空白或伪造结果；
9. 无效输入只阻断相关分支；
10. 其他独立分支继续计算。

五、v0.1 必须包含的反向找条件能力

v0.1 必须支持用户指定目标变量，并由系统反向分析还缺少哪些条件。

系统应：

1. 从目标变量反向搜索可用公式路径；
2. 按路径方案显示缺失条件；
3. 区分已经满足的条件和仍缺少的条件；
4. 区分 Missing、Invalid、Conflict、Model not selected、Assumption disabled、Applicability condition not met；
5. 显示每条路径的模型名称、适用条件和假设；
6. 对不同物理模型路径进行明确标注；
7. 推荐下一步最值得补充的输入；
8. 不伪造数值推导；
9. 不对未登记方向的公式进行自动代数反解。

反向找条件是 v0.1 必须能力，不推迟到后续版本。

六、v0.1 不做自动任意代数反解

v0.1 不允许系统自动对任意公式进行代数变形。

只有公式系统明确登记了某个输出方向，系统才允许按该方向计算或反向找条件。

例如：

P = T × ω

只有当公式系统明确登记后，才允许：

1. 已知 T 和 ω，求 P；
2. 已知 P 和 ω，求 T；
3. 已知 P 和 T，求 ω。

没有登记的输出方向不得自动生成。

七、v0.1 必须包含的结果展示能力

v0.1 必须支持：

1. 主要结果展示；
2. 其他推导结果展示；
3. 中间结果默认折叠；
4. 结果状态标签；
5. 冲突结果优先显示；
6. 强警告结果优先显示；
7. 过期结果标记 Needs recalculation；
8. 用户输入值默认不重复进入普通结果区；
9. 用户输入值与推导值需要比较时，在结果区显示对比；
10. 每个结果默认突出一个主单位；
11. 其他单位或 SI 内部值展开查看；
12. 无结果时显示明确提示，而不是空白页面。

八、v0.1 必须包含的推导展示能力

v0.1 必须支持每个结果展开推导详情。

推导详情至少显示：

1. 推导摘要；
2. 使用公式；
3. 变量代入；
4. 单位换算；
5. 中间变量；
6. 用户输入来源；
7. 推导来源；
8. 系统常数；
9. 简化假设；
10. 警告、冲突或过期状态；
11. 多路径结果的每条路径推导；
12. 无效路径的未完成推导说明。

v0.1 使用树状结构或步骤列表展示推导链，不要求复杂图谱。

九、v0.1 必须包含的默认假设能力

v0.1 必须支持已确认的默认简化假设。

系统应：

1. 默认启用规定的简化假设；
2. 假设必须可见；
3. 只突出显示实际参与当前计算的假设；
4. 推导详情中列出具体假设；
5. 用户可以关闭单项假设；
6. 用户可以关闭全部默认假设；
7. 用户可以恢复默认假设；
8. 用户输入或推导出真实值后，真实值替代假设；
9. 假设关闭后，如果没有真实值替代，对应路径进入缺失条件状态。

十、v0.1 必须包含的警告和合理性能力

v0.1 必须区分四类状态：

1. 正常；
2. 异常但可能；
3. 明显不合理；
4. 无效或无法计算。

v0.1 至少支持：

1. 基础物理无效值检查；
2. 数学无效检查；
3. 分母为零检查；
4. 明显单位误用提示；
5. 候选单位建议；
6. 用户确认保留异常值；
7. User confirmed warning 标记；
8. 无效输入只阻断相关分支；
9. 推导结果异常时显示警告；
10. 警告不因用户确认而消失。

十一、v0.1 必须包含的多路径能力

v0.1 必须支持：

1. 用户输入值与推导值并存；
2. 用户输入值默认作为 Active；
3. 推导值用于核对；
4. 显示绝对差值；
5. 显示百分比差值；
6. 不静默覆盖结果；
7. 同模型路径冲突标记 Conflict；
8. 只要存在同模型冲突，整体状态优先为 Conflict；
9. 不同物理模型结果不标记为普通冲突；
10. 不同模型结果分别展示；
11. Active、Alternative、Conflict、Stale、Invalid 等状态可显示；
12. 过期结果不参与后续推导。

v0.1 不要求完成所有复杂模型优先级，但数据结构必须支持后续扩展。

十二、v0.1 公式覆盖原则

Part 2 不直接确认 v0.1 的具体公式编号和完整公式清单。

v0.1 公式范围应在第三阶段“公式与变量系统”重新确认。

Part 2 只确认公式覆盖原则：

v0.1 只需要覆盖一个小而完整的公式模块。

该模块应满足：

1. 能形成多步推导链；
2. 涉及单位换算；
3. 涉及默认假设；
4. 涉及至少一个中间变量；
5. 能测试反向找条件；
6. 能测试缺失条件路径；
7. 能测试用户输入值与推导值比较；
8. 能测试异常值和单位误用；
9. 能测试多路径或不同模型展示；
10. 能完整验证已确认的 Part 2 产品规则。

v0.1 不以公式数量作为完成标准，而以规则闭环是否成立作为完成标准。

十三、v0.1 必须具备的变量元数据

v0.1 不得依赖变量名称硬编码展示逻辑。

变量元数据至少应支持：

1. variable_id；
2. name；
3. symbol；
4. dimension；
5. default_unit；
6. allowed_units；
7. display_role；
8. importance；
9. normal_range；
10. warning_range；
11. invalid_range。

这些字段可以在后续阶段继续扩展，但 v0.1 的方向必须是元数据驱动。

十四、v0.1 必须具备的公式元数据

公式元数据至少应支持：

1. formula_id；
2. name；
3. expression；
4. inputs；
5. output；
6. allowed_output_direction；
7. model_group；
8. model_name；
9. assumptions；
10. applicability_conditions；
11. priority 或 primary_path。

具体公式内容和字段细节在第三阶段“公式与变量系统”确认。

十五、v0.1 必须具备的结果对象字段

结果对象至少应支持：

1. variable_id；
2. value_si；
3. display_value；
4. display_unit；
5. source；
6. status；
7. display_role；
8. formula_path；
9. dependencies；
10. assumptions_used；
11. model_name；
12. is_active；
13. is_stale；
14. warnings。

结果对象用于运行时展示和推导追踪，不得只保存最终数字。

十六、v0.1 必须通过的验收场景：正常正向计算

用户输入一组足够条件后，系统应：

1. 算出当前可得结果；
2. 显示主要结果；
3. 显示其他推导结果；
4. 折叠中间结果；
5. 展开后可查看公式、代入、单位换算、来源和假设。

十七、v0.1 必须通过的验收场景：缺少关键变量

用户输入不足时，系统应：

1. 算出当前能算的部分结果；
2. 对无法得到的重要目标显示缺失条件；
3. 按路径方案说明还缺什么；
4. 推荐下一步补充输入；
5. 不把无效、冲突或模型未选择错误地显示为普通 Missing。

十八、v0.1 必须通过的验收场景：目标反向找条件

用户指定目标变量但没有足够输入时，系统应：

1. 显示目标变量；
2. 反向列出可行路径；
3. 显示每条路径所需条件；
4. 显示已经满足的条件；
5. 显示缺少的条件；
6. 标注不同物理模型路径；
7. 不伪造数值结果。

十九、v0.1 必须通过的验收场景：单位误用

用户输入数值在当前单位下明显不合理，但换成另一个支持单位后合理时，系统应：

1. 先提示可能选错单位；
2. 显示推荐单位；
3. 显示使用推荐单位后的物理含义；
4. 不自动修改单位；
5. 用户接受后，相关结果标记为需要重新计算；
6. 用户保留原值后，警告继续存在。

二十、v0.1 必须通过的验收场景：无效输入只阻断相关分支

当用户输入无效值时，系统应：

1. 标记该值无效；
2. 不允许该值参与计算；
3. 只阻断依赖该值的分支；
4. 其他独立分支继续计算；
5. 缺失条件区说明该值是 Invalid，不是 Missing。

二十一、v0.1 必须通过的验收场景：用户输入值与推导值并存

当用户输入某个变量，而该变量也能被公式推导时，系统应：

1. 保留用户输入值；
2. 保留推导值；
3. 用户输入值默认 Active；
4. 推导值作为校验；
5. 显示绝对差值；
6. 显示百分比差值；
7. 不静默覆盖用户输入值；
8. 用户选择改用推导值后，相关结果进入待重新计算状态。

二十二、v0.1 必须通过的验收场景：假设参与计算

当某个结果使用默认简化假设时，系统应：

1. 正常计算；
2. 显示 Uses assumptions；
3. 展开后列出具体假设；
4. 显示假设来源 Assumed；
5. 用户关闭假设后，相关结果进入缺失条件状态；
6. 如果用户输入或推导真实值，真实值替代假设。

二十三、v0.1 必须通过的验收场景：结果过期

用户修改输入后，系统应：

1. 找出受影响结果；
2. 将受影响结果标记 Needs recalculation；
3. 保留旧值供用户参考；
4. 过期结果不再参与后续计算；
5. 不受影响结果继续有效；
6. 点击 Recalculate 后更新结果和推导；
7. 如果重算后无法得到旧结果，移出有效结果区并转入缺失条件说明。

二十四、v0.1 明确不做

v0.1 不做：

1. 用户账号；
2. 云端保存；
3. 数据库；
4. 后端服务；
5. 多车辆对比；
6. 历史记录；
7. 图表曲线；
8. PDF 导出；
9. 自动完整报告；
10. OCR 识别题目；
11. 自动代数变形所有公式；
12. 覆盖整本公式册；
13. 复杂 developer/debug mode；
14. 移动端深度适配；
15. 多语言完整切换；
16. 自定义仪表盘；
17. 结果收藏。

v0.1 应保持纯前端、可部署、可验证。

二十五、v0.1 完成标准

v0.1 完成不是指页面能算出一个数，而是必须同时满足：

1. 已确认的 Part 2 产品规则在最小公式范围内可运行；
2. 数据结构不阻断后续扩展；
3. 一个完整公式模块能完成正向推导；
4. 用户指定目标时能反向找条件；
5. 缺失条件能按路径显示；
6. 结果能分层展示；
7. 推导能展开查看；
8. 单位系统能正常工作；
9. 警告和合理性检查能正常工作；
10. 多路径结果不静默覆盖；
11. 输入修改后能正确标记过期结果；
12. 无效输入只阻断相关分支；
13. 默认假设可见、可关闭、可被真实值替代；
14. 没有明显硬编码导致后续公式系统无法扩展；
15. 可以部署到 GitHub Pages。

二十六、v0.1 交付标准

v0.1 交付时至少应包含：

1. 可运行的前端页面；
